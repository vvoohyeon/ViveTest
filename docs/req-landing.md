## 1) 목표 요약 (3~6줄)
- 랜딩페이지는 “다양한 테스트/콘텐츠 카탈로그”를 카드 그리드로 제공하고, 사용자는 카드 탐색(스크롤) 중 즉시 흥미를 느끼며 진입할 수 있어야 한다.
- 카드에는 실시간 조명/tilt로 “생동감”을 제공하되, 텍스트 가독성과 멀미/피로도를 해치지 않도록 각도/속도/감쇠를 제한한다.
- 카드 hover/tap 시 flip으로 “추가 정보(예: 테스트 첫 문항, 블로그 요약, 태그)”를 노출해 클릭 전 의사결정을 돕는다.
- 반응형(Desktop/Tablet/Mobile)에서 그리드가 자연스럽게 재배치되며, i18n 문자열 길이 변화에도 레이아웃이 깨지지 않아야 한다.
- 일부 카드는 unavailable(coming soon)로 노출될 수 있으며, 시각적으로 구분되되 탐색 흐름을 방해하지 않는다.
- 이번 범위(버전 1)에서는 배경 요소(background elements)는 미구현 또는 정적(업데이트 0) 으로 둘 수 있다. 이 경우 본 문서의 배경 관련 항목은 ‘강도 0/정지’로 간주하며, 카드/콘텐츠 인터랙션 요구사항의 구현·QA를 우선한다. 추후 확장 시에만 배경 규칙을 활성화한다.

## 2) 화면 구조 명세 (레이아웃)

### 2.1 컨테이너 규칙
- Page container:
  - max-width: 1280px
  - align: center (좌우 자동 여백)
  - side padding:
    - Desktop/Tablet: 24px (기본), 단 좁아질수록 20px까지 허용
    - Mobile: 16px 고정
- Vertical rhythm:
  - GNB 아래 콘텐츠 시작 spacing: 16~24px (Viewport 높이/밀도에 따라 clamp)
  - 섹션 간격: 24~40px (과한 섹션 분리는 없고, 카드 그리드 중심)

### 2.2 브레이크포인트 정의 (2~3개)
- Mobile: 0 ~ 767px
- Tablet: 768 ~ 1023px
- Desktop: 1024px 이상

> 업계 표준 범위로 제안하며, 실제 구현은 `min-width` 기반으로 단순화 권장.

### 2.3 GNB 명세
- 구성요소:
  - 좌측: CI(로고) (클릭 시 홈)
  - 우측: 햄버거 메뉴 버튼(탭/클릭 영역 44x44px 이상)
  - 페이지 컨텍스트별 구성(필수 고정):
    - Desktop / 랜딩 페이지:
      - CI(홈) + 메뉴(테스트 이력, 블로그 링크 등) 전체 나열
    - Desktop / 테스트 페이지:
      - CI + Timer Count + 테스트 이력 등(노출 정보 최소화)
    - Mobile / 랜딩 페이지:
      - CI + 햄버거 메뉴
    - Mobile / 테스트 페이지:
      - 뒤로가기 + Timer Count
    - Mobile / 블로그 페이지:
      - 뒤로가기 + 햄버거 메뉴

- 레이아웃/동작:
  - Sticky: top: 0
  - Height:
    - Desktop/Tablet: 64px
    - Mobile: 56px
  - Background: 불투명(또는 반투명 + blur는 선택) / 스크롤 시에도 텍스트 대비 유지
  - Shadow: 스크롤 시작(> 4px) 시 얕은 그림자 on (없으면 구분선 1px)
  - z-index: 1000 이상 (카드 hover 확대/flip 오버레이보다 낮지 않게 주의: hover 카드가 GNB 위로 올라가면 안 됨)
  - 전환 중 GNB 교체 타이밍(필수 고정):
    - 전환 중에는 GNB를 유지하고, 전환 완료 시점에 한 번에 목적 페이지 GNB로 교체한다.
    - 요소 구성이 달라지는 경우(예: Mobile 블로그 페이지의 ‘뒤로가기&햄버거’)도 동일 원칙을 적용한다.

- 접근성:
  - 햄버거 버튼: aria-label 필수
  - 포커스 링: 키보드 탐색 시 명확히 표시

### 2.4 카드 그리드 명세

#### 2.4.1 카드 간격(16~20px) 해석 규칙
- grid gap = clamp(16px, 1.5vw, 20px)
- 카드 내부 padding = 16px (기본), 정보 밀도가 높을 경우 14~18px 범위에서 조절 가능
- 카드 radius:
  - 기본: 16px
  - Mobile full-bleed 모드(아래 3.3): radius 축소(0~12px 범위) 가능

#### 2.4.2 Desktop: 2/3/4 컬럼 “조합” 규칙 (구현 가능 규칙으로 고정)
> “첫 줄 2~3, 이후 3~4”를 **2단 그리드(히어로 행 + 본문 그리드)**로 구현한다.

- 공통:
  - 카드 리스트를 `heroCards`(상단 2~3개) + `mainCards`(나머지)로 분리 렌더링
  - `availableWidth` 는 Page container 내부에서 좌우 패딩이 적용된 이후의 실제 콘텐츠 영역 너비를 의미한다.
- hero row(첫 줄):
  - 조건:
    - availableWidth(컨테이너 실제 너비) >= 1160px 이면 heroCols = 3
    - 그 외 heroCols = 2
  - heroCards 개수:
    - heroCols=3이면 상단 3개
    - heroCols=2이면 상단 2개
- main grid(나머지):
  - 조건:
    - availableWidth >= 1160px 이면 mainCols = 4
    - 그 외 mainCols = 3
- 카드 최소 폭:
  - Desktop: minCardWidth 260~300px 범위(가독성/정보량 기준으로 팀 내 확정)
- 정렬:
  - 동일 row 내 카드 높이는 “콘텐츠 높이”가 아니라 “카드 기본 비율(예: 4:5~3:4)”을 우선(텍스트는 line-clamp 처리)

#### 2.4.3 Tablet: 2/3 컬럼 “조합” 규칙
- 동일하게 `heroCards` + `mainCards` 2단 구조 유지
- hero row(첫 줄): heroCols = 2 고정, heroCards = 상단 2개
- main grid(나머지):
  - availableWidth >= 900px 이면 mainCols = 3
  - 그 외 mainCols = 2
- 본문 내 중복 정의로 인한 충돌을 방지하기 위해, 텍스트 줄수/말줄임은 **2.5.4(텍스트 줄수/말줄임 정책)**을 단일 기준으로 따른다.

#### 2.4.4 Mobile: 1 컬럼 + 좌우 16px 여백
- single column list
- 컨테이너 side padding: 16px 고정
- 카드 간 vertical gap: 14~16px (공간 과다 방지)
- 카드 높이: 콘텐츠에 따라 auto
- 텍스트 줄수/말줄임(필수 고정)
  - 앞면(Front)
    - title: 1줄 고정, overflow 시 말줄임(truncate)
    - subtitle: 1줄 고정, overflow 시 말줄임(truncate)
    - tags: 0~3개, 각 태그 라벨은 1줄 고정 + 말줄임(truncate)
    - availabilityStatus/배지: 1줄 고정 + 말줄임(truncate)
  - 뒷면(Back)
    - 테스트 카드: 질문 1개(최대 2줄) + 선택지 2개(각 최대 2줄) — overflow 시 말줄임(truncate)
    - 블로그 카드: 요약 최대 4줄 — overflow 시 말줄임(truncate)

### 2.5 카드 앞/뒷면 정보: “데이터 슬롯” 및 노출 규칙 정의
> 카드 타입(테스트/블로그/기타)에 따라 동일 슬롯 구조를 재사용한다.  
> 본 절은 “뒷면 정보”뿐 아니라, **앞면(Front)과 뒷면(Back)의 노출 매핑**까지 포함한다.  
> 슬롯이 정의되어도, **해당 면(Front/Back)에 노출 대상으로 지정되지 않으면 표시하지 않는다.**

#### 2.5.1 카드 면(Front/Back) 기본 정의
- Front(앞면): 카탈로그 탐색 시 기본으로 노출되는 면
- Back(뒷면): flip으로 노출되는 추가 정보 면 (클릭/탭 전 의사결정 보조)
- 공통 원칙:
  - Front에는 **진입(CTA) 요소를 두지 않는다**(flip 전 CTA 없음).
  - Back에는 카드 타입별로 정의된 정보/CTA만 노출한다(Front의 정보가 Back에 “자동 복제”되지는 않는다. 단, Unavailable 테스트 카드는 예외 규칙을 따른다).

#### 2.5.2 카드 타입별 Front 노출 슬롯(필수/선택)
- 테스트 카드 Front(필수 고정)
  - `cardTitle` (필수)
  - `cardSubtitle` (필수)
  - `thumbnailOrIcon` (필수)
  - `tags[]` (0~3개 노출)
  - `availabilityStatus` = {available | unavailable}
- 블로그 카드 Front(필수 고정)
  - `cardTitle` (필수)
  - `cardSubtitle` (필수)
  - `thumbnailOrIcon` (필수)
  - `tags[]` (0~3개 노출)
  - `availabilityStatus` = {available}  // 블로그 카드는 Unavailable 상태로 존재하지 않음(아래 2.5.6)

#### 2.5.3 카드 타입별 Back 노출 슬롯(필수/선택) + CTA 정의
- Back 헤더(공통, 필수 고정)
  - `cardTitle`만 노출한다.
  - Back 헤더의 `cardTitle`은 Front에 표시한 `cardTitle`과 **동일한 값으로 고정**한다(불일치 금지).
  - Back 헤더에는 `cardSubtitle`, `thumbnailOrIcon`, `tags[]`를 노출하지 않는다(타이틀만 유지).

- 테스트 카드 Back(available)
  - `previewQuestion` (필수): “첫 문항 질문” 1개
    - 줄수: 최대 2줄(overflow 시 말줄임)
  - `answerChoiceA` / `answerChoiceB` (필수): 첫 문항의 2개 선택지
    - 줄수: 각 최대 2줄(overflow 시 말줄임)
  - 의미 고정(필수):
    - 각 선택지는 Back에서 노출되는 **진입 CTA**이며, 선택 즉시 다음을 만족해야 한다.
      1) 선택한 값이 **테스트 Q1 응답으로 사전 확정(pre-answered)** 된다.
      2) 테스트 진입 직후 화면은 **Q2부터 시작**한다(Q1은 이미 응답된 상태로 간주).
    - 사용자는 테스트 진행 중 **이전 문항으로 이동**하여 Q1을 **수정할 수 있다**.
    - 분석/결과의 기준은 **“최종 제출된 Q1”**만을 사용한다(랜딩에서의 최초 선택은 기준값이 아님).
    - `previewQuestion` 및 `answerChoiceA/B`는 **본 테스트의 Q1과 동일 소스**를 사용하며, 내용 불일치가 발생하는 케이스는 **존재할 수 없다(불일치 금지)**.
  - `meta` (필수, 3개 고정):
    - 메타데이터1: 예상 소요 시간
    - 메타데이터2: 공유 횟수
    - 메타데이터3: 누적 테스트 횟수
  - 테스트 카드 Back에는 “Start” 같은 별도 `primaryCTA`를 두지 않는다(선택지 2개가 CTA 역할을 대체).

- 블로그 카드 Back(available)
  - `summary` (필수): 블로그 내용 요약
    - 줄수: 최대 4줄(overflow 시 말줄임)
  - `meta` (필수, 3개 고정):
    - 메타데이터1: 읽는 데 걸리는 시간
    - 메타데이터2: 공유 횟수
    - 메타데이터3: 조회수
  - `primaryCTA` (필수, 1개 고정):
    - 라벨: “Read more”로 고정(i18n 적용 대상)
    - Back에서만 노출되는 진입 CTA

#### 2.5.4 텍스트 줄수/말줄임(truncate) 정책(필수 고정)
- 공통 원칙:
  - 줄바꿈으로 레이아웃이 늘어나는 대신, **말줄임(truncate)**을 우선 적용한다.
  - 본 문서 단계에서는 “무엇을 먼저 희생할지(우선순위로 삭제/비노출)” 정책을 확정하지 않고, **각 슬롯별 말줄임 규칙**만 고정한다.
- Front(테스트/블로그 공통)
  - `cardTitle`: 1줄 고정 + 말줄임
  - `cardSubtitle`: 1줄 고정 + 말줄임
  - `tags[]`: 0~3개, 각 태그 라벨 1줄 고정 + 말줄임
  - `availabilityStatus`/배지 텍스트: 1줄 고정 + 말줄임
- Back(카드 타입별)
  - 테스트 카드:
    - `previewQuestion`: 최대 2줄 + 말줄임
    - `answerChoiceA/B`: 각 최대 2줄 + 말줄임
    - `meta`: 단일 행/블록 내에서 overflow 발생 시 말줄임(표현 방식은 자유, “정보 3개 유지”가 원칙)
  - 블로그 카드:
    - `summary`: 최대 4줄 + 말줄임
    - `meta`: 단일 행/블록 내에서 overflow 발생 시 말줄임
    - `primaryCTA`: 1줄 고정 + 말줄임(라벨은 “Read more” 고정이므로 일반적으로 짧게 유지)

#### 2.5.5 슬롯 누락/숨김 처리 정책(대안 3가지 제안)
> 목적: 운영 데이터가 비정상일 때(또는 선택 슬롯이 비어있을 때) UI가 깨지지 않고 QA에서 빠르게 탐지되도록 한다.

- 공통 전제(필수 고정):  
  - “필수”로 정의된 슬롯이 누락되는 경우는 **오류(데이터/QA 이슈)**로 간주한다.  
  - 이 경우 “없던 것으로 숨김 처리”하지 않고, **누락된 상태로 렌더링(빈 값/공백 영역 포함)**하여 QA에서 발견 가능하게 한다.
  - 필수 슬롯 누락 노출 정책은 QA/스테이징 환경에서 우선 적용한다. 운영 환경에서는 레이아웃 붕괴를 막는 범위에서 최소한의 방어 정책을 허용할 수 있다(단, ‘없던 것으로 숨김 처리’ 금지 원칙을 침해하지 않는 선에서).

- 선택 슬롯(`tags[]` 등) 처리:
  - 선택 슬롯 자리 유지(플레이스홀더/빈 영역 유지)
    - 선택 슬롯이 비어도 레이아웃 골격은 유지하고, 값만 비어 보이게 둔다.

#### 2.5.6 Unavailable 카드 콘텐츠/인터랙션 고정 규칙

- 블로그 카드:
  - Unavailable 상태로 존재하지 않는다(coming soon 블로그 카드 금지).

- 테스트 카드(availabilityStatus=unavailable):
  - 공통 원칙(필수 고정):
    - 진입 CTA는 제공하지 않는다(테스트 진입 불가).
    - flip(Front↔Back 전환)은 사용자 입력으로 발생하지 않는다(Desktop/Mobile/키보드 공통).
    - “Coming soon” 안내가 최우선으로 읽히되, 카드의 기본 정보(타이틀/서브타이틀/썸네일/태그/상태)는 희미하게라도 인지 가능해야 한다.

  - Front(필수 고정):
    - 테스트 카드 Front 슬롯 구성(2.5.2)은 유지한다.
    - Front 위에 “Coming soon” 안내 오버레이를 표시한다(i18n 적용).

  - Desktop 동작(필수 고정):
    - hover enter 후 120~200ms 유지 시:
      - “Coming soon” 오버레이를 표시한다(스침 방지).
      - HOVER_LOCK(Desktop)을 발동한다(3.0 정의 따름).
    - hover out 시:
      - 오버레이를 해제하고, HOVER_LOCK(Desktop)을 해제한다(해제 시점/지연은 3.0 정의 따름).
    - 주의: Unavailable 테스트 카드는 Desktop에서 Back이 “사용자에게 노출되는 상태”로 전환되지 않는다(= flip 금지).

  - Mobile 동작(필수 고정):
    - 기본 상태(디폴트)에서 “Coming soon” 오버레이가 항상 표시된다(hover 전제 없음).
    - 탭 입력은 피드백을 제공하지 않는다(= flip 토글 없음, full-bleed 진입 없음, 안내 재표시/강조 등 추가 반응 없음).

  - 키보드 포커스 동작(필수 고정):
    - Unavailable 테스트 카드에 포커스는 허용한다.
    - 포커스 시:
      - “Coming soon” 오버레이를 표시한다(시각 안내).
      - HOVER_LOCK(Desktop)을 발동한다(Desktop 환경에서만, 3.0 정의 따름).
    - 포커스 상태에서도 진입 CTA는 제공하지 않으며, Back 콘텐츠 접근(선택지/메타 등)은 허용하지 않는다.

  - 오버레이 시각 규칙(필수 고정):
    - 검정색 오버레이 레이어 위에 흰색 “Coming Soon” 메시지를 표시한다.
    - 오버레이는 메시지 가독성을 최우선으로 하되, 카드 정보가 ‘희미하게라도’ 보이도록 처리한다(완전 차단 금지).
    - 포커스 링/키보드 탐색 시각 표시가 오버레이에 의해 식별 불가해지지 않도록 보장한다.

  - 방어 규칙(필수 고정):
    - 어떤 이유로든 Back 면이 노출되는 경우가 발생하더라도(예외 상황),
      - 질문/선택지/메타 등 Back 전용 정보는 노출하지 않고,
      - Front 정보 유지 + “Coming soon” 오버레이만 표시한다.

#### 2.5.7 테스트 카드 seamless continuation(Back 응답 → 본 테스트 이어가기) 계약 (필수 고정)

- 정의:
  - “Seamless continuation”은 랜딩 카드 Back에서의 선택이 **본 테스트 Q1 응답으로 연결**되고, 사용자는 추가 단계 없이 **본 테스트 흐름을 즉시 이어가는 것**을 의미한다.

- 진입/진행 규칙:
  - Back에서 `answerChoiceA/B`를 선택하면, 선택 값은 **Q1 응답으로 사전 확정(pre-answered)** 되며,
    테스트 진입 직후 화면은 **Q2부터 시작**한다.
  - Q1은 테스트 내에서만 수정 가능하며, 사용자는 **이전 문항으로 이동하여 수정**할 수 있다.

- 정합성/기준값 규칙:
  - 통계/분석/결과 산출에서 Q1은 **“최종 제출된 Q1”**만을 기준으로 삼는다(랜딩 최초 선택은 기준값이 아님).
  - `previewQuestion` 및 `answerChoiceA/B`는 본 테스트 Q1과 동일 소스를 사용하므로, 불일치 케이스는 존재할 수 없다(불일치 금지).

- 실패/취소(롤백) 규칙:
  - 전환이 실패/취소된 경우, Q1 사전 확정은 **롤백(없던 일)** 처리되며,
    “테스트 시도”는 성립한 것으로 간주하지 않는다(부분 성립/부분 기록 금지).

- 재진입 규칙:
  - 테스트 페이지에서 뒤로가기로 랜딩 복귀 후, 동일 카드에서 재선택하면 **항상 새 테스트 시도**로 간주한다(이전 시도 재개 금지).

- 문항 체류시간(백그라운드 포함) 집계 규칙:
  - 문항별 체류시간은 포그라운드 여부와 무관하게 **경과 시간 전체**를 포함한다.
  - 동일 문항 재방문이 발생할 경우, 해당 문항 체류시간은 **총 누적 시간(재방문 포함 합산)**으로 집계한다.
  - 이상치 처리(지표 산출 시):
    - 테스트 시도 내 문항 체류시간 중 **가장 긴 상위 5%**는 “이상치 후보”로 간주한다.
    - 단, 상위 5%와 나머지 구간의 **편차가 충분히 크지 않은 경우**에는 제외하지 않고 포함한다.
    - “편차가 충분히 큰지”의 판정 기준은 운영 파라미터로 별도 고정한다(문서 내 수치 미확정).

### 3.0 공통 상태 정의 (Page/Card)
- PageState:
  - `ACTIVE` (입력 신호 정상)
  - `INACTIVE` (브라우저 focus out/visibility hidden)
  - `REDUCED_MOTION` (prefers-reduced-motion)
  - `SENSOR_DENIED` (모바일 자이로 권한 거부/미지원)
  - `TRANSITIONING` (페이지 진입 전환 진행 중: 입력/스크롤 잠금 및 시각 상태 고정)

- CardState(각 카드별):
  - `REST` (기본)
  - `TILTING` (tilt 적용 중)
  - `FLIPPED` (뒷면 노출)
  - `FOCUSED` (키보드 포커스/접근성)

- PageState 전역 반응 규칙 (상태 전이 시 1회 적용)
  - PageState=`INACTIVE` 진입 시
    - 반응:
      - 카드 상호작용 입력(hover/tap/키보드 기반 상태 전이)과 “입력 기반 업데이트(tilt/조명 반응)”를 중지한다.
      - 모든 카드는 시각 상태를 “읽기/가독성 우선” 기준으로 정리한다(FLIPPED 유지/해제 여부는 하위 규칙을 따르되, tilt/조명 반응은 중지).
      - `HOVER_LOCK(Desktop)`은 비활성으로 간주한다(전역 입력 억제 원칙 우선).
    - 제약/우선:
      - `REDUCED_MOTION`이 활성인 경우, INACTIVE 동안 배경을 포함한 모든 모션은 정지(또는 최소화)한다.
      - INACTIVE 동안 허용되는 움직임이 있다면 “배경 레이어의 저주파 idle drift”로 제한하며, 콘텐츠(카드/텍스트)에는 적용하지 않는다.
  - PageState가 `ACTIVE`로 복귀하는 시점
    - 반응:
      - 입력 기반 반응은 즉시 100%로 복귀하지 않고, 짧은 적응 구간(램프업) 후 정상 반응으로 복귀한다(급격한 변화 방지).
  - PageState=`TRANSITIONING` 진입 시(필수 고정)
    - 반응:
      - 페이지 스크롤 및 입력 기반 상태 전이(hover/tap/키보드 기반 카드 상태 전이)를 잠금 처리한다.
      - 입력 기반 업데이트(tilt/조명 반응)는 0으로 간주하고, 전환 시작 프레임의 시각 상태를 고정한다.
      - 전환 중(TRANSITIONING)에는 배경 레이어를 포함한 모든 업데이트를 정지(고정 프레임 유지) 한다.
      - GNB는 전환 중 유지하며, 전환 완료 시점에 목적 페이지 구성으로 한 번에 교체한다(2.3 참조).
    - 제약/우선:
      - `INACTIVE` 또는 `REDUCED_MOTION`이 활성인 경우에는 해당 상태의 원칙이 우선하며, 전환 연출은 그 정책을 침해하지 않는 범위로 축소/단순화한다(4항 참조).
- GlobalOverride(전역 규칙/플래그)
  - `HOVER_LOCK(Desktop)`
    - 의미:
      - Desktop에서 “단 1개의 카드가 읽기/주의 집중 상태”일 때,
        나머지 카드의 입력 반응을 억제하여 읽기/주의 집중을 보장하는 전역 규칙.
      - “읽기/주의 집중 상태”에는 다음이 포함된다:
        1) available 카드가 hover flip으로 `FLIPPED` 상태에 진입한 경우
        2) Unavailable 테스트 카드가 hover(120~200ms 유지)로 “Coming soon” 오버레이가 활성화된 경우
        3) Unavailable 테스트 카드가 키보드 `FOCUSED` 상태에서 “Coming soon” 오버레이가 활성화된 경우
    - 트리거:
      - Desktop 환경에서 위 조건 중 하나가 성립하는 즉시(각 조건의 지연/발동 규칙은 하위 섹션 정의를 따른다).
    - 반응(필수 고정):
      - 대상 카드(현재 읽기/주의 집중 상태의 카드)를 제외한 모든 카드는 `REST`로 강제한다.
      - `HOVER_LOCK` 활성 동안, 비대상 카드의 “입력 기반 반응(tilt/조명 등)”은 0으로 간주한다.
      - `HOVER_LOCK` 활성 동안, 비대상 카드에서 hover/flip 시도 자체도 허용하지 않는다(의도치 않은 상태 전이 금지).
    - 해제:
      - 대상 카드의 읽기/주의 집중 상태가 종료되는 시점에 해제한다.
        - available: hover out로 `FLIPPED` 종료 시 해제(해제 지연 등은 하위 규칙을 따른다)
        - Unavailable hover: 오버레이 해제 시 해제(해제 지연 등은 하위 규칙을 따른다)
        - Unavailable focus: 포커스 이탈 시 해제(포커스 이동으로 새로운 대상이 생기면 대상만 교체된다)
    - 제약/우선:
      - `PageState=REDUCED_MOTION` 또는 `PageState=INACTIVE` 상태에서는 `HOVER_LOCK(Desktop)`을 적용하지 않는다(모션 억제/입력 비활성 원칙 우선).
      - `HOVER_LOCK(Desktop)`은 CardState가 아니라 “전역 덮어쓰기 규칙”이며, 비대상 카드의 상태를 단일 기준으로 강제하기 위해 존재한다.

> 우선순위(상위가 하위를 덮음):
`INACTIVE` > `REDUCED_MOTION` > `TRANSITIONING` > `FLIPPED` > `HOVER_LOCK(Desktop)` > `TILTING` > `REST`

> 참고: `FOCUSED`는 키보드 접근성 관점의 상태이며, 시각/모션 상태(`FLIPPED`/`TILTING`/`REST`)와의 결합 가능 여부 및 제약은 각 하위 규칙에서 명시한다.


### 3.1 공통 효과

#### 3.1.1 조명 효과(광원) 정의: 추상 파라미터
- Light model(추상):
  - `lightPos` = (x: -1~+1, y: -1~+1, z: 0~1)  // 화면 정규화 좌표
  - `lightColor` = rgba(255,255,255, α=0.08~0.20)  // 과도한 하이라이트 방지
  - `intensity` = 0.3~0.8
  - `falloff` = 0.6~0.9  // 중심에서 가장자리로 감쇠
- 적용 대상:
  - background elements(선택/추후 확장): 점과 라인만으로 구성된 장식 레이어(정보 비전달).
  - 순서: GNB > 카드(콘텐츠) > 배경. 배경은 카드/텍스트 가독성에 어떠한 영향도 주면 안 된다(대비/밝기/가독성 변화 금지).
  - 카드 표면(고주파, 빠른 반응) — 단 텍스트 영역은 대비 유지(아래 4 참고)

#### 3.1.2 입력 신호
- Desktop:
  - pointer position (x,y): `pointermove`
  - scroll: `scrollY`, `scrollVelocity`
- Mobile:
  - gyroscope/device orientation: `beta/gamma` 기반 tilt
  - scroll: `scrollVelocity`
- 배경 레이어 입력 우선순위(추후 확장 시): 포인터 입력 > 단말 기울기 입력 > (둘 다 없을 때) 의도적인 랜덤 반응 순으로 적용한다. 단, 배경 반응은 항상 저주파(읽기 방해 금지)로 제한한다.

#### 3.1.3 카드 Tilt 규칙 (가독성 제한 포함)
- Desktop tilt(마우스 상대 위치):
  - 트리거: `pointermove` (단, 현재 FLIPPED 카드가 없고 HOVER_LOCK이 없을 때)
  - 반응: 카드 중심 대비 포인터 상대 위치로 tilt 계산
  - 파라미터(권장 범위):
    - maxTiltDeg = 6° ~ 10° (기본 8°)
    - responseSmoothing = 80ms ~ 180ms (저역통과/스프링 감쇠)
    - updateRate = rAF 기반, 단 60fps 유지 어려우면 30fps로 degrade
- Mobile tilt(기울기 기반):
  - 트리거: deviceorientation (권한 허용 시)
  - 반응: (gamma, beta)를 정규화해 tiltX/tiltY로 매핑
  - 파라미터(권장 범위):
    - sensorClamp = ±15° 입력 제한
    - maxTiltDeg = 5° ~ 9° (기본 7°)
    - smoothingHalfLife = 120ms ~ 220ms
- 가독성 보호 방식 선택(필수 고정)
  - 기본값: (A) 텍스트/아이콘 레이어는 별도 평면(또는 독립 처리)으로 간주하고, tilt/조명 영향은 50% 이하로 제한한다.
  - 대안: (B) 텍스트 레이어 분리를 사용하지 않는 경우에만, 텍스트 대비 자동 보정을 적용한다(그라데이션/오버레이 강도 상한을 유지).
  - 혼용 금지: 한 카드에서 (A)와 (B)를 동시에 적용하지 않는다(중복 보정으로 인한 과보정/가독성 저하 방지).
  - 우선순위: `REDUCED_MOTION`에서는 (A)/(B) 모두 “최소 움직임/최소 변화” 원칙에 맞춰 단순화한다(읽기 우선).

#### 3.1.4 카드 Flip 규칙
- Desktop:
  - available 카드:
    - 트리거: `hover enter` 후 120~200ms 유지 시 flip (의도 없는 스침 방지)
    - 반응: 앞면→뒷면 flip (또는 crossfade는 버전 A에서)
  - Unavailable 테스트 카드:
    - 트리거: `hover enter` 후 120~200ms 유지 시 “Coming soon” 오버레이 표시 (의도 없는 스침 방지)
    - 반응:
      - flip은 발생하지 않는다(Front 유지).
      - 오버레이 활성화와 동시에 `HOVER_LOCK(Desktop)`을 적용한다(3.0 참조).
    - 해제: hover out 시 오버레이 해제 + HOVER_LOCK 해제(해제 지연/안정화는 아래 Desktop 공통 해제 규칙을 따른다).

- Mobile:
  - available 카드:
    - 트리거: `tap` 시 flip 토글
    - 반응: 앞면↔뒷면 전환 + (선택) full-bleed 모드 진입
  - Unavailable 테스트 카드:
    - 기본 상태에서 “Coming soon” 오버레이가 노출된다(2.5.6 참조).
    - 탭은 flip 토글을 수행하지 않으며, 추가 피드백을 제공하지 않는다.

- 전환 시작점(필수 고정):
  - 전환은 FLIPPED 상태에서 Back의 CTA가 활성화되는 순간에 시작한다.
    - 테스트 카드: `answerChoiceA/B` 중 1개 선택(선택 즉시 진입)
    - 블로그 카드: “Read more” CTA 클릭/탭(클릭 즉시 진입)

- 전환 중 입력/스크롤 정책(필수 고정):
  - 전환 시작과 동시에 페이지 스크롤 및 카드/배경 입력 기반 반응(hover/tap/포인터 반응/센서 반응 포함)을 잠금 처리한다.
  - 전환 중에는 FLIPPED/확대 상태가 사용자 입력(hover out 등)으로 되돌아가지 않도록 “현재 시각 상태를 고정”한다.

  - 전환 중 뒤로가기 입력 정책(필수 고정):
    - 전환 진행 중 사용자의 뒤로가기 입력(브라우저/OS back 포함)은 **즉시 처리하지 않고 1회 큐잉**한다.
    - 큐잉된 뒤로가기 입력은 전환이 **완료된 직후 1회만 반영**한다.
    - 전환이 실패/취소되어 롤백된 경우, 큐잉된 뒤로가기 입력은 **반영하지 않고 폐기**한다(중복 이탈/중복 상태전이 방지).

- 뒤로가기(복귀) 시 복원 범위(필수 고정):
  - 테스트/블로그 페이지에서 뒤로가기로 랜딩 복귀 시, 반드시 복원해야 하는 것은 (a) 그리드 스크롤 위치(scrollY)만이다.
  - 아래 항목은 복원을 요구하지 않으며, 랜딩 복귀 시 기본값으로 정리한다:
    - 직전 카드의 FLIPPED 유지 여부: 복원하지 않음(기본 REST)
    - Desktop 120% 상태 복원: 복원하지 않음
    - 키보드 포커스 복원: 복원을 요구하지 않음(기본 포커스 정책은 접근성 원칙을 훼손하지 않는 선에서 구현에서 결정)

- Desktop “특정 카드가 읽기/주의 집중 상태일 때 나머지 카드 입력 억제”:
본 규칙은 3.0의 HOVER_LOCK(Desktop) 정의를 따른다(트리거/반응/해제/우선순위 중복 정의 금지). 아래는 요약이며, 상충 시 3.0이 단일 기준이다.
  - 트리거:
    - available 카드가 `FLIPPED`(hover flip) 상태가 되는 즉시
    - Unavailable 테스트 카드에서 “Coming soon” 오버레이가 활성화되는 즉시
  - 반응:
    - 그 외 카드들은 `REST`로 강제(입력 반응 억제는 3.0의 HOVER_LOCK 정의를 따른다).
  - 해제:
    - 대상 상태 종료(available: hover out으로 FLIPPED 종료 / Unavailable: 오버레이 해제) 시 해제
    - 해제 시점에는 깜빡임 방지를 위한 짧은 지연을 허용한다(지연 범위는 120~200ms로 제한).

- Flip vs Tilt/조명 충돌 우선순위:
  - FLIPPED 카드(available):
    - tilt: 0°로 고정 또는 최대 2°로 축소(선택)  ← “읽기” 우선
    - 조명: 강도 70%로 축소(텍스트 대비 유지)
  - Unavailable 오버레이 카드(Desktop hover / 키보드 focus):
    - flip 없음(Front 유지)
    - 오버레이 가독성을 우선하며, 카드의 입력 기반 반응은 읽기 방해가 되지 않도록 최소화한다.
  - 비대상 카드:
    - HOVER_LOCK 미적용 시 tilt/조명 정상
  - reduced motion:
    - tilt off, 조명 off(또는 약화), flip은 fade/slide로 대체

- Back 콘텐츠/CTA(카드 타입별, 필수 고정)
  - Back 헤더: 타이틀만 노출(2.5.3 참조)
  - Desktop - 테스트 카드(available):
    - flip 전 CTA 없음
    - flip 후 첫 문항의 2가지 답변 선택지가 CTA이며, 선택 즉시 테스트 진입 + Q1 답변으로 확정(2.5.3 참조)
  - Desktop - 블로그 카드(available):
    - flip 전 CTA 없음
    - flip 후 `primaryCTA`는 “Read more”로 고정(2.5.3 참조)
  - Unavailable 테스트 카드:
    - 진입 CTA는 제공하지 않는다.
    - flip이 발생하지 않으며, 오버레이만 제공한다(2.5.6 참조).

- Card navigation(진입)은 FLIPPED 이후 CTA로만 가능(필수 고정)
  - Desktop:
    - available 카드: hover flip으로 뒷면이 열린 상태에서만 CTA로 진입
    - 카드(Front/Back)의 CTA 외 영역 클릭은 진입 트리거가 아니다
    - Unavailable 카드: 진입 불가(CTA 없음, flip 없음)
  - Mobile:
    - available 카드: tap은 flip 토글만 수행(첫 탭 진입 금지), 진입은 뒷면 CTA로만 수행
    - Unavailable 카드: 탭 피드백 없음, 진입 불가(CTA 없음, flip 없음)

- 키보드 접근성(필수 고정, CTA 접근 경로 보장)
  - available 카드:
    - 키보드 탐색으로 카드가 `FOCUSED` 상태가 되면, Desktop의 hover flip과 동등하게 Back(뒷면) 콘텐츠에 접근 가능해야 한다(CTA가 Back에만 존재하기 때문).
    - 키보드로는 Back에 노출된 CTA 요소(테스트: 선택지 2개 / 블로그: Read more)만 진입 트리거가 된다.
    - 키보드 탐색으로 카드가 FOCUSED가 되었을 때, 사용자는 명시적인 키보드 조작(예: 활성화 키 혹은 스페이스바) 으로 Back 상태로 진입할 수 있어야 하며, Back에 진입한 이후에만 CTA(선택지/Read more)로 이동·활성화가 가능해야 한다. (FOCUSED만으로 자동 flip시키는 방식은 선택 사항이며, 선택 시에도 의도치 않은 flip을 막기 위한 지연/해제 규칙을 동일하게 적용한다.)
  - Unavailable 테스트 카드:
    - 포커스는 허용하되, 진입 CTA는 제공하지 않는다.
    - 포커스 시 “Coming soon” 오버레이를 시각적으로 제공한다(2.5.6 참조).
    - Desktop 환경에서는 포커스 시 `HOVER_LOCK(Desktop)`을 동일하게 적용한다(3.0 참조).
  - 전환 중 키보드/포커스 안정성(필수 고정):
    - 키보드로 CTA를 활성화한 경우에도 전환 중 포커스 표시/탐색이 붕괴하지 않도록 보장한다(포커스 링이 “사라져 보이지 않음”을 금지).

### 3.2 Desktop 전용

#### 3.2.1 Browser Focus Out 시: 배경 요소 느리게 움직임
- 트리거: `window blur` 또는 `document.visibilityState !== 'visible'`
- 반응: background elements에 “idle drift” 적용
- 파라미터(추상):
  - driftAmplitude = 6~14px
  - driftPeriod = 12~24s
  - driftEasing = ease-in-out
- 제약:
  - 카드(콘텐츠) 자체는 drift 금지(가독성/멀미 방지), 배경 레이어만 적용

#### 3.2.2 마우스 진입 직후 0.5초: 배경 반응 게인을 빠르게 램프업(초기 가속 구간)
- 트리거: `pointerenter` 발생
- 반응:
  - 0~0.5s: gain 0.3 → 1.0 램프업(초기 적응 구간)
  - 0.5s 이후: gain = 1.0 유지(일반 반응 속도로 정착)
- 파라미터:
  - rampDuration = 500ms
  - rampEasing = cubic-bezier(0.2, 0.8, 0.2, 1)

#### 3.2.3 Hover로 flip 된 카드는 120% 확대
- 적용 대상(필수 고정):
  - Desktop에서 available 카드가 hover flip으로 `FLIPPED`에 진입한 경우에만 적용한다.
  - Unavailable 테스트 카드는 hover 시 flip이 발생하지 않으므로, 본 확대 규칙을 적용하지 않는다.

- 트리거: (available) 카드가 Desktop hover로 `FLIPPED` 진입 시
- 반응:
  - 120% 확대
  - 확대 기준점(transform-origin)은 카드 위치에 따라 화면 유효 영역 침범을 피하도록 조정될 수 있다.
  - 120% 확대의 목적/지위(필수 고정):
    - 본 120% 확대는 “hover 상태에서 Back을 읽기 쉽게 하기 위한 연출(읽기용)”이다.
    - FLIPPED 유지 동안 120% 상태는 유지되며, hover out 또는 “페이지 진입 전환 시작” 전까지 유지된다.

- 레이아웃 깨짐 방지 규칙:
  - 확대는 레이아웃에 영향을 주지 않는 방식으로 처리한다(리플로우 금지).
  - FLIPPED 카드의 레이어 우선순위는 GNB보다 낮게 유지한다(카드가 GNB 위로 올라가면 안 됨).

- 잘림/겹침 방지: 자동 보정(필수)
  - 목표: 확대(120%) 시 카드가 “뷰포트 유효 영역”을 침범해 잘리거나, GNB/화면 경계와 충돌하지 않게 한다.
  - 뷰포트 유효 영역(권장): 
    - 상단: GNB 하단 기준 여백 8~16px 확보
    - 좌/우/하단: 화면 경계 기준 여백 8px 이상 확보
  - 자동 보정 우선순위(상위가 하위를 우선 적용):
    1) 확대 기준점을 카드 위치에 따라 조정해 화면 안쪽으로 확장되도록 유도
    2) 위 조정으로도 침범이 발생하면, 확대 비율을 “허용 범위 내에서” 완화한다(120% 고정이 불가능한 케이스의 예외 규칙)
    3) 매우 좁은 뷰포트 등으로 위 규칙을 모두 적용해도 침범이 해소되지 않으면, 해당 카드에 한해 확대를 생략하고 FLIPPED만 유지한다(가독성/조작성 우선)
  - 좌표/크기 기억 기준(필수 고정):
    - “페이지 진입 전환”에 사용할 카드의 좌표/크기는, 120% 확대 및 자동 보정(기준점 조정/스케일 완화/확대 생략 등)이 모두 반영된 “최종 결과(화면에 실제로 보이는 외곽)”를 기준으로 한다.
    - 즉, 보정으로 카드가 미세 이동/스케일 완화된 케이스에서도 ‘보정 이후 결과’를 기억한다.
  - 안정성 제약:
    - 자동 보정은 FLIPPED 진입 시 1회 결정 후 hover 유지 동안 고정한다(프레임마다 흔들리는 보정 금지).
    - 페이지 진입 전환이 시작되는 순간, 확대 비율/기준점/자동 보정 결과는 전환 완료까지 고정한다(전환 중 재계산/흔들림 금지).
    - 전환 시작 이후에는 hover out 등 입력 변화로 인해 120% 상태가 해제되어 “되돌아가는” 현상을 금지한다.

### 3.3 Mobile 전용

#### 3.3.1 기울기 감지로 tilt
- 트리거: deviceorientation permission granted
- 반응: 각 카드 tilt 적용(단, 화면에 보이는 카드만)
- 제약:
  - 화면 내 visible 카드만 업데이트(Intersection ratio >= 0.25)

  - 스크롤 상태 정의(모바일 공통)
    - “사용자 드래그 스크롤 중”: 사용자가 화면을 터치한 채 스크롤 제스처를 수행하는 동안
    - “관성 스크롤 중”: 사용자가 손을 뗀 이후에도 화면이 계속 이동하는 동안
    - “정지”: 스크롤 이동이 사실상 멈춘 상태

  - 스크롤 상태에 따른 센서 tilt gain 조정(멀미 방지)
    - 사용자 드래그 스크롤 중: 센서 tilt gain을 50%로 감소
    - 관성 스크롤 중: 센서 tilt gain은 즉시 100%로 복귀하지 않고, 짧은 안정화 구간 후 점진 복귀(급격한 변화 방지)
    - 정지: 센서 tilt gain 100% 적용

#### 3.3.2 스크롤 속도에 따른 tilt (간단 함수/구간)
- 트리거: scroll event로 velocity 계산(v = ΔscrollY / Δt, px/ms)
- 반응: 추가 tiltBoostDeg 적용(스크롤 방향에 따른 “살짝 눌림/젖힘”)
- 매핑(예시, 구현 고정 규칙):
  - |v| < 0.5 → tiltBoost = 0°
  - 0.5 ≤ |v| < 1.5 → tiltBoost = lerp(0°, 3°)
  - |v| ≥ 1.5 → tiltBoost = 5° cap
- 결합 규칙:
  - 최종 tilt = gyroTilt * 0.7 + scrollTiltBoost * 0.3 (기본)
  - FLIPPED 상태에서는 scrollTiltBoost = 0° (읽기 우선)

#### 3.3.3 Flip 된 카드는 full-bleed 표시(좌우 16px 여백 제거)
- 트리거: Mobile에서 카드 tap → `FLIPPED` 진입
- 반응(기본: in-flow full-bleed)
  - 카드 폭: 100vw
  - 좌우 확장: 컨테이너 좌우 패딩을 “상쇄”하여 edge-to-edge가 되도록 한다(구현 방식은 자유, 결과만 고정).
  - full-bleed 전환 애니메이션:
    - duration: 220~360ms
    - easing: cubic-bezier(0.2, 0.8, 0.2, 1)

- full-bleed 상태의 스크롤/콘텐츠 정책
  - 페이지 진입 전환과의 관계(필수 고정):
    - Mobile에서 ‘진입’은 FLIPPED + full-bleed 상태에서 Back CTA를 통해서만 발생한다(3.1.4의 진입 정의를 따른다).
    - 페이지 진입 전환은 “full-bleed로 확장된 FLIPPED 카드의 최종 외곽(화면에 실제로 보이는 크기/위치)”을 출발점으로 삼는다.
  - page scroll lock: full-bleed 상태에서는 페이지 스크롤을 잠근다(배경 탐색 방지, 포커스/조작성 안정화 목적).
    - “긴 텍스트 처리”는 아래 2안 중 1개만 허용하며, 혼용 금지:
    버전 B 기본값은 in-flow full-bleed + (안1) line-clamp + 더보기로 고정한다. 오버레이형 full-bleed 및 내부 스크롤(안2)은 버전 C에서만 허용한다(버전 간 혼용 금지).
      - (안 1) line-clamp + ‘더보기’ (기본값)
        - 카드 내부 스크롤은 사용하지 않는다.
      - (안 2) 카드 내부 스크롤
        - 내부 스크롤을 허용하는 경우 ‘더보기’는 제공하지 않는다(중복 패턴 금지).
        - 내부 스크롤은 “오버레이형 full-bleed(외부 닫기 영역/백드롭이 존재)”에서만 허용한다.
    - 선택 기준(명시):
      - in-flow full-bleed에서는 기본값(안 1)만 사용한다.
      - 오버레이형 full-bleed에서만 안 2를 허용한다.

- 스크롤/복귀/닫기 규칙
  - 진입 시(선택): 해당 카드가 화면 상단(예: GNB 아래 8~16px)에 오도록 위치를 맞춘다.
    - 제약: 위치 맞춤을 사용하는 경우, page scroll lock은 “위치 맞춤이 완료된 뒤” 적용한다(충돌 방지).
  - 전환 중 입력/스크롤 정책(필수 고정, 3.1.4와 동일):
    - Back CTA로 페이지 진입 전환이 시작되는 순간, 스크롤/입력은 잠금 처리한다.
    - 전환 시작 이후에는 ‘닫기(재탭/외부 탭/뒤로가기)’ 규칙이 페이지 진입 전환과 충돌하지 않도록, 전환 완료(또는 취소) 전에는 상태가 흔들리지 않게 고정한다.
  - 닫기:
    - 재탭(같은 카드) → 닫힘
    - 카드 외부 탭으로 닫기(조건부)
      - 외부 탭 닫기는 “카드 바깥의 탭 가능한 외부 영역”이 실제로 존재하는 경우에만 지원한다.
      - in-flow full-bleed에서는 외부 탭 영역이 없으므로 기본 미지원으로 본다.
      - 오버레이형 full-bleed에서는 외부 탭 닫기를 권장한다.
    - OS back 제스처/뒤로가기 → 닫힘(라우팅 변경이 없으면 상태만 복원)
  - 닫힐 때: full-bleed 해제 → 원래 컨테이너 폭 + 좌우 16px 복귀

- 제약:
  - full-bleed 상태에서 다른 카드 상호작용 비활성 + page scroll lock

### 3.4 텔레메트리/이벤트 로깅 요구사항(UX + 데이터 계약) (필수 고정)

- 기본 방침(범위 최소화):
  - 버전 1의 로깅 목적은 “랜딩 → 테스트/블로그 진입 전환”의 안정성과, 테스트 완료/제출 지표를 최소 이벤트로 측정하는 것이다.
  - 카드 탐색 성격의 상호작용(스크롤/hover/flip/tap 토글/tilt/조명 등)은 **기본 미수집**으로 한다.
  - Unavailable 카드에 대한 hover/tap “시도”는 **기록하지 않는다(미수집)**.

- 익명 개인 식별(동일 브라우저/동일 기기 범위):
  - 식별 목표 범위는 **동일 브라우저/동일 기기 내 일관된 묶음**이면 충분하다(사실상 device/browser 단위).
  - 금지(필수): IP 주소, 브라우저/기기 특성 조합(지문) 등 외부 식별 가능성이 있는 값으로 개인을 추정/구성하지 않는다.
  - 권장(필수 고정): 브라우저 내에 저장되는 **무작위 기반의 익명 식별자 1종**을 사용한다.
    - 수명: **브라우저 데이터가 유지되는 한 장기 유지**(사용자 삭제/초기화 전까지).

- 동의/옵트아웃(Consent) 정책:
  - 기본 수집, 사용자 거부 가능(옵트아웃 제공).
  - 옵트아웃 시:
    - 익명 식별자 및 “식별 가능성이 있는 연결키 계열”은 **즉시 무효화**하며 이후 생성/전송하지 않는다.
    - 집계성 이벤트는 유지하되, 특정 사용자/기기/세션에 **연결될 수 없어야 한다**(개인 추적 불가).

- 이벤트 최소 세트(필수 고정):
  - Landing:
    - 페이지 진입(1회)
  - Transition(테스트/블로그 공통):
    - 전환 시작(1회)
    - 전환 완료(1회) 또는 전환 실패/취소(1회) — 상호 배타
  - Test:
    - 테스트 시도 시작(1회)
    - 테스트 최종 제출(1회)

- TRANSITIONING과 로깅의 정합성(필수 고정):
  - 3.1.4의 “전환 시작점”에서 전환 시작 이벤트를 1회 기록한다.
  - TRANSITIONING 동안 입력은 잠금되므로, 동일 전환에 대해 시작 이벤트가 중복 기록되지 않아야 한다.
  - 전환 결과는 완료/실패(취소 포함) 중 하나로만 종료되며, 실패 시에는 롤백 규칙(2.5.7)과 정합되어야 한다.

- 테스트 응답/문항 체류시간 로깅(필수 고정):
  - 수집 금지: 문항/선택지의 **원문 텍스트**, 자유입력, 개인정보 등 식별 가능 데이터.
  - 수집 원칙: “테스트/문항/선택지”는 **의미 코드**로만 기록한다(예: 선택지 A/B 같은 의미 구분).
  - 문항 인덱스는 **1-based**로 고정한다.
  - 테스트 최종 제출 시점에 다음을 기록한다(시도 단위):
    - **최종 제출된 응답**(문항별 선택지)
    - **문항별 체류시간**(총 누적 시간, 재방문 포함, 백그라운드 포함)
  - Q1은 랜딩에서 사전응답되더라도, 통계/분석/결과 기준은 **“최종 제출된 Q1”**만 사용한다(랜딩 최초 선택은 기준값이 아님).

- 체류시간 이상치 처리(지표 산출 규칙, 필수 고정):
  - 테스트 시도 내 문항 체류시간 중 **가장 긴 상위 5%**는 “이상치 후보”로 간주한다.
  - 단, 상위 5%와 나머지 구간의 **편차가 충분히 크지 않은 경우**에는 제외하지 않고 포함한다.
  - “편차가 충분히 큰지”의 판정 기준은 운영 파라미터로 별도 고정한다(문서 내 수치 미확정).

## 4) 접근성/성능 최소 가이드 (과하지 않게, 랜딩 범위만)
- prefers-reduced-motion:
  - tilt: off
  - 조명: off 또는 intensity를 0.2 이하로 약화
  - flip: 3D 회전 대신 crossfade/slide(150~200ms)로 대체
- 페이지 진입 전환(landing → 테스트/블로그)에서의 reduced motion 규칙(필수 고정):
  - 기본값: 페이드 중심 + 아주 짧은/작은 변화(예: 약한 확대 또는 짧은 이동감 중 1개만)로 최소 연속성을 제공한다.
  - 금지: 카드 외곽 좌표/크기를 끝까지 추적하며 화면을 채우는 “큰 확대 연출”을 reduced motion에서 강제하지 않는다.
  - 저사양/성능 저하 감지 시에는 ‘완전 페이드 중심(공간 이동/확대 연출 없음)’으로 자동 단순화할 수 있다(가독성/멀미 방지 우선).

- 저사양/권한 미허용(graceful fallback):
  - Mobile에서 센서 권한 거부/미지원 시:
    - gyroTilt = 0
    - scrollTiltBoost만 약하게(최대 2~3°) 적용하거나 완전 off(버전 A)
  - Desktop에서 성능 저하 감지 시:
    - 배경 조명/패럴럭스 업데이트 주기를 30fps로 낮추고, 카드 업데이트는 visible 카드만 유지
- 텍스트 가독성 원칙(2~4개)
  - (1) 텍스트 대비는 항상 확보: 조명 하이라이트 alpha 상한(≤0.20), 어두운 오버레이 최소 보장
  - (2) FLIPPED 상태에서는 tilt/조명을 반드시 약화(읽기 우선)
  - (3) 텍스트 줄수/말줄임은 “2.5.4 텍스트 줄수/말줄임(truncate) 정책”을 단일 기준으로 적용한다(브레이크포인트별 중복 정의로 인한 충돌 금지).
  - (4) Back의 진입 CTA는 텍스트 가독성을 우선하며, CTA가 말줄임되더라도 “CTA 영역/상호작용 가능”은 명확히 구분되어야 한다.

## 5) 개선안 2~3버전 제시 (필수)
버전 A/B/C는 서로 배타적인 전체 패키지이며, 한 번 선택되면 문서 1~4의 관련 파라미터/규칙은 선택한 버전으로 치환한다. 본 문서 본문(1~4)은 기본값으로 버전 B 기준으로 작성되었으며, 다른 버전을 선택할 경우 해당 섹션의 규칙을 버전 정의에 맞춰 적용한다.

### 버전 A (보수적/구현 쉬움)
- 레이아웃(그리드/브레이크포인트)
  - BP: Mobile≤767 / Tablet 768~1023 / Desktop≥1024
  - Desktop: heroCols=2 고정, mainCols=3 고정(4컬럼 없음)
  - Tablet: hero=2, main=2(3컬럼 없음)
  - Mobile: 1컬럼 + 16px 패딩
- 조명/tilt/flip 우선순위 규칙
  - 조명: 카드에만 약하게(intensity 0.3~0.5), 배경 조명 없음
  - tilt: Desktop pointer 기반만(최대 6~8°), Mobile tilt off(센서/스크롤 모두 off)
  - flip: Desktop hover(지연 150ms) / Mobile tap(토글)
  - FLIPPED 시 tilt=0, 조명=50%로 축소
- Mobile full-bleed 카드 처리
  - full-bleed 미적용(컨테이너 16px 유지)
- 왜 좋은가(2~3줄)
  - 구현 리스크가 가장 낮고, 성능/접근성 이슈가 적다.
  - “flip을 통한 정보 제공” 핵심만 확보하며, 나머지 효과는 단계적으로 확장 가능.

### 버전 B (균형형/권장)
- 레이아웃(그리드/브레이크포인트)
  - BP 동일
  - Desktop: 2단 그리드
    - heroCols= (width>=1160 ? 3 : 2)
    - mainCols= (width>=1160 ? 4 : 3)
  - Tablet: hero=2, main= (width>=900 ? 3 : 2)
  - Mobile: 1컬럼 + 16px 패딩
- 조명/tilt/flip 우선순위 규칙
  - 조명: 배경(저주파) + 카드(고주파) 모두 적용, intensity 상한 준수(0.3~0.8)
  - tilt:
    - Desktop pointer 기반(max 8°)
    - Mobile gyro + scrollBoost 결합(gyro 0.7 / scroll 0.3, max 7° + boost 5° cap)
  - flip:
    - Desktop hover flip + FLIPPED 카드 scale 1.2
    - Desktop에서 FLIPPED 동안 다른 카드 tilt=0 강제
    - Mobile tap flip
  - 충돌 처리: FLIPPED 시 tilt 0~2°로 축소 + 조명 70%로 축소(읽기 우선)
- Mobile full-bleed 카드 처리
  - in-flow full-bleed(100vw + 패딩 상쇄) 적용
  - 닫기: 재탭 + 뒤로가기(상태 복원)
- 왜 좋은가(2~3줄)
  - 원 의도(조명/tilt/flip + 모바일 full-bleed)를 대부분 보존하면서도, 우선순위/제약으로 구현 가능성을 확보한다.
  - 성능/접근성 fallback 규칙이 명확해 QA 범위를 통제할 수 있다.


### 버전 C (연출 강화/복잡)
- 레이아웃(그리드/브레이크포인트)
  - BP 동일 + Desktop에서 width>=1160 시 mainCols=4 고정
  - hero row는 3개 고정(가능하면), 첫 row 카드에 “featured” 스타일(높이/비율 차등) 추가
- 조명/tilt/flip 우선순위 규칙
  - 조명: 배경 레이어에 다중 요소(2~3개) drift + 포인터/자이로 반응(게인 동적)
  - tilt: 카드별 “시차(phase)”를 약간 두어 군집이 살아 움직이는 느낌(단, maxTilt 상한은 유지)
  - flip:
    - Desktop hover 시 scale 1.2 + 주변 dim 처리(배경/타 카드 brightness -10~ -20%)
    - Mobile tap 시 full-bleed + 반투명 backdrop(외부탭 닫기 강제)
  - 우선순위: backdrop 활성 시 다른 카드 입력 완전 차단(모달 준수)
- Mobile full-bleed 카드 처리 방식
  - 오버레이형 full-bleed(사실상 bottom-sheet/overlay):
    - 페이지 스크롤 lock
    - 카드 내부 스크롤 허용
    - 닫기: 외부탭/닫기 버튼/뒤로가기 모두 지원
- 왜 좋은가(2~3줄)
  - 몰입감/주목도는 가장 높고, “상호작용하는 랜딩”의 인상이 강하다.
  - 다만 입력 차단/스크롤 lock/오버레이 전환 등으로 구현·QA 복잡도와 접근성 리스크가 증가한다(특히 iOS 권한/스크롤 충돌).
